# "明理" - 新一代AI命理智能推演平台规范

## 一、项目背景

### 现有项目分析

**Tianfu Agent** 是目前市场上较为成熟的AI命理分析工具，其核心特点包括：

1. **技术架构**
   - 多Agent协作架构：紫微斗数、子平八字、奇门遁甲三大体系各有专属Agent
   - 可视化推理流程：每一步分析透明可追溯
   - 长期个性化记忆系统：Profile-based架构，支持跨会话追踪
   - 内部排盘引擎：避免外部数据冲突，保证结果一致性

2. **功能模块**
   - 250+定制化命盘工具
   - 专业钦天派紫微斗数方法论
   - 不确定性综合分析模型（八字与紫微斗数交叉验证）
   - 多维度人生解析：性格、事业、财运、婚恋、健康等十多个维度

3. **商业模式**
   - 免费额度：0次/天
   - 付费订阅：$9.9/10次问题
   - 目标用户：有一定认知深度的命理爱好者及专业从业者

4. **核心优势**
   - 解决了通用AI排盘不稳定、大运计算错误、解读过于教条等问题
   - 飞星、四化推演错误率远低于通用大模型
   - 推理过程可视化，增强了用户信任度

### 现有不足

1. **扩展性限制**：三大体系耦合度高，难以灵活扩展其他命理流派
2. **知识封闭**：缺乏开放的知识网络和案例共享机制
3. **API能力弱**：面向开发者的接口不完善，难以二次开发
4. **协作能力缺失**：缺乏多用户协作、专业复核机制
5. **移动端体验**：主要为Web端，移动端适配待优化

## 二、为什么需要新项目

### 问题与机会

**核心矛盾**：
- 准确性 vs 可解释性：用户既需要准确的结果，又需要理解决策过程
- 专业深度 vs 易用性：专业用户需要深度工具，新手需要简单引导
- 效率 vs 全面性：快速响应与深度分析需要平衡
- 个性化 vs 标准化：个性化建议与可复现的标准化流程

**市场机会**：
- AI命理市场正处于从"玩具级"向"专业级"跃迁的窗口期
- 垂直类工具相比通用AI具有显著的专业优势
- 用户对推理透明性和可验证性的要求日益提高

## 三、项目概述

### 项目名称

**"明理"（MingLi AI）** —— 明命理、知进退

### 核心定位

"明理"是一个开放、可扩展的AI命理智能推演平台，不仅是一个分析工具，更是一个命理知识操作系统。

### 核心价值主张

1. **推理透明**：每一步分析都有据可查，结论可验证
2. **知识开放**：整合古籍原典、现代研究、实践案例
3. **生态协作**：用户反馈持续优化，专业人士可参与复核
4. **技术领先**：模块化架构，支持灵活扩展和深度定制

### 核心模块设计

#### 1. 排盘引擎 (Charting Engine)

**职责**：统一管理各类命盘的生成和计算

**功能**：
- 支持紫微斗数、子平八字、奇门遁甲、七政四余等主流命理体系
- 内部排盘，确保结果一致性
- 真太阳时校准
- 支持批量排盘

**API接口**：
```
POST /api/v1/chart/ziwei
POST /api/v1/chart/bazi
POST /api/v1/chart/qimen
POST /api/v1/chart/convert
```

#### 2. 推理引擎 (Reasoning Engine)

**职责**：执行命理分析和推理

**功能**：
- 规则匹配与演绎推理
- 多体系交叉验证
- 不确定性量化表达
- 推理路径可视化

**核心算法**：
- 基于规则的专家系统
- 贝叶斯网络处理不确定性
- 知识图谱支持复杂推理

#### 3. Agent系统 (Multi-Agent System)

**架构**：Orchestrator + 领域Agent

**Orchestrator职责**：
- 任务理解与分解
- Agent调度与协调
- 结果整合与输出
- 质量控制与验证

**领域Agent**：
- ZiweiAgent：紫微斗数专项分析
- BaziAgent：子平八字专项分析
- QimenAgent：奇门遁甲专项分析
- GeneralAgent：综合分析与建议生成

#### 4. 记忆系统 (Memory System)

**架构**：短期记忆 + 长期记忆 + 跨会话记忆

**短期记忆**：
- 当前会话上下文
- 最近的分析记录
- 实时状态

**长期记忆**：
- 用户Profile
- 历史分析摘要
- 个人特质标签

**跨会话记忆**：
- 命盘关键变化追踪
- 重大事件记录
- 预测结果回溯验证

#### 5. 知识库 (Knowledge Base)

**古籍库**：
- 《紫微斗数全书》《渊海子平》《奇门遁甲秘笈》等原典
- 带章节索引和关键概念标注

**规则库**：
- 各流派核心规则结构化存储
- 规则优先级和适用条件
- 规则版本管理

**案例库**：
- 真实命例结构化存储
- 分析过程和结论记录
- 结果反馈和验证

**范式库**：
- 经典格局模式库
- 常见问题解答模式
- 专业术语解释

## 六、数据模型

### 6.1 核心实体

```python
# 命盘基础信息
Profile:
  - id: UUID
  - name: str
  - birth_datetime: datetime
  - birth_location: Location
  - gender: Gender
  - timezone: str
  - created_at: datetime
  - updated_at: datetime

# 命盘数据
Chart:
  - id: UUID
  - profile_id: UUID
  - chart_type: ChartType  # ZIWEI, BAZI, QIMEN
  - data: dict  # 结构化命盘数据
  - generated_at: datetime

# 分析记录
Analysis:
  - id: UUID
  - profile_id: UUID
  - chart_id: UUID
  - analysis_type: str
  - query: str
  - reasoning: ReasoningChain  # 推理链
  - result: AnalysisResult
  - confidence: float  # 置信度
  - created_at: datetime

# 推理链
ReasoningChain:
  - steps: List[ReasoningStep]
  - sources: List[Source]  # 引用来源
  - alternative_paths: List[ReasoningPath]  # 备选推理路径

# 知识条目
KnowledgeEntry:
  - id: UUID
  - category: str
  - title: str
  - content: str
  - source: Source
  - tags: List[str]
  - related_entries: List[UUID]
```

### 6.2 关系图

```
User (用户)
  ├── Profile (命盘档案) 1:N
  │     ├── Chart (命盘) 1:N
  │     │     ├── Analysis (分析) 1:N
  │     │     │     └── ReasoningChain (推理链)
  │     │     └── Event (事件记录) 1:N
  │     └── Feedback (反馈) 1:N
  └── Subscription (订阅) 1:1
```

## 九、需求规范

### 9.1 新增需求

#### 需求：模块化命理引擎

系统 SHALL 提供可插拔的命理引擎架构，支持动态加载和卸载不同的命理体系（紫微斗数、子平八字、奇门遁甲等）。

**场景：添加新命理体系**
- **WHEN** 开发者需要添加七政四余体系
- **THEN** 系统提供标准化的插件接口，开发者按照规范实现后即可无缝集成

#### 需求：推理可视化

系统 SHALL 提供完整的推理链路可视化，包括每一步的依据、规则引用和备选路径。

**场景：查看分析依据**
- **WHEN** 用户查看分析结果
- **THEN** 系统展示完整的推理流程图，每个节点可点击查看详细依据

#### 需求：不确定性表达

系统 SHALL 提供量化的置信度表达，不使用简单概率，而是基于多维度因素综合评估。

**场景：评估结论可靠性**
- **WHEN** 用户需要判断结论的可靠性
- **THEN** 系统展示置信度等级（高/中/低）及其影响因素

#### 需求：开放API

系统 SHALL 提供完整的RESTful API，支持第三方应用调用核心功能。

**场景：开发者集成**
- **WHEN** 开发者需要将命理功能集成到自己的应用
- **THEN** 系统提供完善的API文档和SDK

#### 需求：知识网络

系统 SHALL 提供可扩展的知识库，支持古籍、规则、案例的整合和检索。

**场景：查询术语解释**
- **WHEN** 用户查询"天机星"的解释
- **THEN** 系统返回知识库中相关的定义、历史来源、经典案例

#### 需求：记忆追踪

系统 SHALL 提供长期记忆能力，记录用户状态变化和预测验证。

**场景：预测回溯**
- **WHEN** 用户记录重大事件
- **THEN** 系统关联历史预测，标注命中或偏差
